Vision for the service. Where it goes

   Hierarchy. As for now service monitors only web elements usage. Stats are flat (not hierarchical) and have no visual comparison to each other (to see which elements are used the most and which the least). That's why next steps are to display hierarchical models of data (for example button "Cancel" is on that page in that place, it has its unique path — so you could know exactly which "Cancel" button in stats) and show elements usage volumes compared to each other (on sankey diagrams).

   Data. Interacting with elements users choose options and enter data. We want to display analysis for the data as well. For example, to answer on questions like: "Which options are most often selected in dropdown list and which never selected?".

   Features usage. Elements here and there — are the parts of more global things — features. Features could be tagged and monitored and analysed as well. Same as elements features are hierarchical items.

   Users. Okey, someone uses that element or feature or enters that data into the input field. Why should we care at all? Maybe it's just QA or PM or UX designer or Sales or Marketing team interacts with your app or service in prod. We want stats to include UID(s) and UGID(s) — so you could filter only users and users groups that matter or exclude users and groups that irrelevant. Don't optimise something just because "someone uses this". Know who uses "this" and optimise for those who matters.
